# 标准开发流程

## 1. 目标

建立一条轻量、可复用、可检查的开发闭环，避免直接进入编码导致返工。

流程如下：

`需求 -> 原型 -> 原型确认 -> 技术方案 -> 方案确认 -> 开发 -> 测试 -> 验收 -> 文档`

## 2. 流程说明

### 2.1 需求

目标：

- 明确为什么做
- 明确做什么
- 明确不做什么
- 明确验收标准

最小产物：

- 一份需求任务卡

### 2.2 原型

目标：

- 把需求从文字转为页面或流程表达
- 提前暴露字段、交互、路径、角色差异问题

最小产物：

- 页面原型、流程图、线框图、交互说明中的任意一种或组合

### 2.3 原型确认

目标：

- 确认业务流程正确
- 确认交互路径正确
- 确认页面字段和操作基本完整

最小产物：

- 已确认的原型评审记录

### 2.4 技术方案

目标：

- 决定模块归属
- 决定接口与数据结构
- 决定配置、权限、缓存、MQ、任务等技术处理方式

最小产物：

- 一份技术方案说明

### 2.5 方案确认

目标：

- 确认实现路径合理
- 确认技术风险可控
- 确认不会明显违反现有架构约束

最小产物：

- 已确认的方案评审记录

### 2.6 开发

目标：

- 按已确认的原型和方案实现功能
- 不在开发过程中随意扩需求

最小产物：

- 代码改动
- 必要配置改动
- 必要脚本或 SQL 变更

### 2.7 测试

目标：

- 验证代码级正确性
- 提前发现编译、依赖、接口、逻辑问题

最小产物：

- 编译结果
- 测试结果
- 问题记录与处理结果

### 2.8 验收

目标：

- 验证用户真实使用流程是否成立
- 验证结果是否符合原型和验收标准

最小产物：

- 验收清单
- 截图、录屏、操作记录中的一种或组合

### 2.9 文档

目标：

- 把改动沉淀为可复用知识
- 降低后续接手和扩展成本

最小产物：

- 文档回写记录

## 3. 推荐职责分工

- 需求提出方：提供目标、背景、优先级、验收标准
- 产品或业务负责人：主导需求和原型确认
- 架构或技术负责人：主导技术方案确认
- 开发负责人：实现、测试、自查
- 验收负责人：按清单验证结果
- 文档维护人：同步更新文档

## 4. 执行原则

- 未确认需求，不进入原型
- 未确认原型，不进入技术方案
- 未确认技术方案，不进入开发
- 未完成测试，不进入验收
- 未完成文档回写，不视为真正完成
