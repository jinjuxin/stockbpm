# 开发流程

标准顺序：

`需求追问策略(按需) -> 需求澄清 -> 业务需求拆解 -> 需求确认图 -> 需求完整性检查 -> 需求冲突检查 -> 非功能需求分析 -> 需求评审结论 -> 原型生成 -> 原型确认 -> 技术方案 -> 开发实现 -> 测试清单 -> 验收清单 -> 文档回写`

## 分层说明

### 1. 需求澄清

解决：
- 为什么做
- 给谁用
- 做什么
- 不做什么
- 怎么验收

### 2. 业务需求拆解

解决：
- 业务场景
- 角色
- 模块
- 核心对象
- 业务字段
- 核心流程
- 第一版范围

### 3. 原型生成

解决：
- 页面清单
- 页面字段
- 按钮和交互
- 页面跳转
- 角色界面差异

### 4. 检查与评审

解决：
- 需求是否完整
- 是否存在规则冲突
- 是否遗漏非功能约束
- 是否具备进入下一阶段的条件

## 需求确认图

用于在原型前确认业务结构，默认采用 Mermaid。

### 1. 小需求

默认产物：
- 需求任务卡
- 必要时补一个简单流程图

### 2. 中等需求

默认产物：
- 需求任务卡
- 业务需求拆解
- 功能结构图
- 核心流程图

### 3. 系统级需求

默认产物：
- 需求任务卡
- 业务需求拆解
- 功能结构图
- 核心流程图
- 状态流转图

### 4. 复杂系统

在系统级需求基础上，可追加：
- 思维导图
- 角色关系图
- 异常流程图

规则：
- 先确认业务结构图，再进入页面原型
- 不让页面原型替代功能结构和流程确认

## 检查与评审阶段

默认产物：
- 需求完整性检查
- 需求冲突检查
- 非功能需求分析
- 需求评审结论

规则：
- 系统级需求默认完整执行
- 小需求可按需裁剪，但至少保留完整性检查
- 评审结论未通过，不进入原型或技术方案

## 特殊路径

- 小范围页面修改，可直接进入原型修改或开发修改
- 影响业务规则、范围、验收标准的变更，必须回到需求阶段
