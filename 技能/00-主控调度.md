# 主控调度

## 适用场景

- 接收用户一句话需求
- 判断当前阶段
- 决定调用哪个技能
- 判断是否需要回退前置阶段

## 输入

- 用户输入
- 当前需求目录内容
- 当前应用或原型状态

## 输出

- 阶段判断
- 技能调用决议
- 是否需要人工确认

## 必查项

- 当前是不是新需求
- 是系统级需求，还是单页面/单模块需求
- 是新系统，还是存量系统新增/变更
- 是需求变更，还是开发修改
- 前置阶段是否已完成

## 分层规则

### 1. 需求阶段

这一阶段解决“业务是什么”，而不是“页面怎么画”。

必须明确：
- 为什么做
- 给谁用
- 做什么
- 不做什么
- 怎么验收
- 业务场景
- 角色
- 核心对象
- 业务字段

### 2. 原型阶段

这一阶段把业务内容落到页面。

必须明确：
- 页面清单
- 每页有哪些字段
- 哪些字段只读
- 哪些按钮
- 页面跳转
- 角色差异

结论：
- 需求阶段拆到“业务字段层”
- 原型阶段拆到“页面字段层”

### 3. 需求确认图阶段

这一阶段用于在原型前确认业务结构是否一致。

默认图形产物：
- 功能结构图
- 核心流程图
- 状态流转图
- 复杂系统可补思维导图

规则：
- 小需求按需出图
- 中等及以上需求默认出功能结构图和核心流程图
- 系统级需求默认再补状态流转图
- 图确认通过后再进入原型阶段

### 4. 检查补强阶段

这一阶段用于把需求从“能看懂”提升到“能继续做”。

默认检查产物：
- 需求完整性检查
- 需求冲突检查
- 非功能需求分析
- 需求评审结论

规则：
- 系统级需求默认执行三类检查
- 存量系统新增默认执行完整性和冲突检查
- 小需求至少按需做完整性检查
- 检查不过，不进入原型或技术方案
- 评审结论未通过，不进入下一阶段

### 5. 需求追问阶段

这一阶段用于处理模糊、范围不清或背景不足的输入。

默认策略：
- 先识别需求类型
- 再按追问策略收敛关键缺口
- 一轮只追问 1 到 3 个关键问题

## 调度规则

### 1. 新的小需求

默认顺序：

`需求追问策略(按需) -> 需求澄清 -> 需求确认图(按需) -> 需求完整性检查(按需) -> 需求评审结论(按需) -> 原型生成 -> 原型评审 -> 技术方案 -> 开发实现`

### 2. 从 0 开始设计一个系统

默认顺序：

`需求追问策略(按需) -> 需求澄清 -> 业务需求拆解 -> 需求确认图 -> 需求完整性检查 -> 需求冲突检查 -> 非功能需求分析 -> 需求评审结论 -> 原型生成 -> 原型评审 -> 技术方案 -> 开发实现`

如果明确要进入可开发状态，还要在方案和开发阶段补充：
- 技术框架搭建
- 目录结构
- 基础组件
- 权限与接口边界

### 3. 存量系统新增/变更

默认顺序：

`需求追问策略(按需) -> 需求澄清 -> 存量系统对齐 -> 需求确认图 -> 需求完整性检查 -> 需求冲突检查 -> 非功能需求分析(按需) -> 需求评审结论 -> 原型生成 -> 原型评审 -> 技术方案 -> 开发实现`

## 禁止事项

- 不跳过阶段门禁
- 不在缺少前置产物时强行进入后续阶段
- 不把存量系统的新增页面做成另一套产品风格
- 不让原型阶段替代需求分析
