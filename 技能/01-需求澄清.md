# 需求澄清

## 作用

- 把用户的原始需求整理成《需求任务卡》
- 在需求阶段提前识别：这是新系统还是存量系统新增/变更
- 在需求阶段明确是否需要搭技术框架
- 为后续原型、方案、开发提供稳定输入

## 适用场景

### 1. 新系统需求

例如：
- 从 0 开始做一个工单系统
- 从 0 开始做一个费用管理系统
- 从 0 开始做一个后台管理平台

此时必须补充：
- 系统目标
- 角色与页面范围
- 是否需要搭建技术框架

### 2. 存量系统新增/变更需求

例如：
- 在已有工单系统上新增“费用申请”
- 在旧页面上增加审批字段
- 在原系统中新增一个列表页

此时必须补充：
- 基于哪个旧系统
- 是否必须继承现有风格
- 参考页面或参考路径是什么

## 必查项

### A. 通用必查项

- 为什么做
- 给谁用
- 做什么
- 不做什么
- 怎么验收

### B. 需求类型必查项

- 是新系统还是存量系统新增/变更
- 是否基于现有系统
- 是否有参考页面、截图、原型或线上地址

### C. 一致性必查项

如果是存量系统新增/变更，必须确认：
- 是否必须保持原系统整体风格一致
- 是否允许局部重设计

默认规则：
- 未明确允许改版时，按“保持一致”处理

### D. 框架必查项

如果是新系统需求，必须确认：
- 这次只是做原型，还是要继续做可开发版本
- 后续是否需要搭建项目技术框架

## 输出要求

需求澄清完成后，输出的《需求任务卡》至少应包含：

- 基本信息
- 背景
- 目标
- 范围
- 用户与场景
- 业务规则
- 影响对象
- 验收标准
- 备注

并额外明确：
- 需求类型：新系统 / 存量系统新增 / 存量系统改造
- 是否基于现有系统：是 / 否
- 是否要求风格一致：必须 / 尽量 / 否
- 是否涉及技术框架搭建：是 / 否

若需求为系统级或中等以上需求，还应继续准备：
- 风险与待确认项
- 后续完整性检查输入
- 后续冲突检查输入

## 缺失信息时的处理

- 只追问关键缺口，不一次性追问过多问题
- 若信息足够支撑下一阶段，可先形成初稿再确认
- 若缺少“需求类型 / 一致性 / 是否搭框架”这类关键判断，不进入原型阶段
- 若存在关键待确认项，不直接宣称需求已完整
- 需求明显模糊时，先交给“需求追问策略”整理追问顺序

## 禁止事项

- 不直接跳到技术方案
- 不跳过验收标准
- 不忽略“新系统 / 存量系统新增”差异
- 不忽略“是否保持旧系统一致”这一关键前提
- 不把待确认项藏在正文里不单独列出
