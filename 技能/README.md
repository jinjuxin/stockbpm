## 使用原则

- 日常使用时，你只需要直接提需求
- 主控负责判断当前阶段，并决定调用哪个技能
- 当需求属于“大系统设计”时，主控会在需求阶段后继续调用“业务需求拆解”
- 当需求属于“原有系统新增/变更”时，主控会调用“存量系统对齐”
- 在进入原型或技术方案前，主控会继续调用完整性检查、冲突检查、非功能分析
- 当需求很模糊时，主控会先调用“需求追问策略”
- 在需求阶段收口前，主控会调用“需求评审结论”
- 当页面进入高视觉要求的原型阶段时，主控应继续调用视觉策略、审美评审、排版、配色和重构能力

## 分层原则

- 需求阶段：明确业务、角色、范围、核心对象和业务字段
- 原型阶段：先明确页面与交互，再补视觉策略、高保真表现、审美评审、排版、配色与重构

也就是说：
- 需求阶段拆到“业务字段层”
- 原型阶段拆到“页面字段层”

## 什么时候显式指定技能

只有在调试流程时，才需要显式指定，例如：

- 只测试“需求澄清”
- 只测试“业务需求拆解”
- 强制按“存量系统对齐”执行
